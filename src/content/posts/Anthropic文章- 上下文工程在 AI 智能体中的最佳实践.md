## 上下文是关键但有限的资源

上下文指的是从大语言模型采样时所包含的一组 token。当前面临的工程问题是，在 LLM 固有约束条件下，优化这些 token 的效用，以持续达成期望的结果。

可以认为，上下文工程是提示词工程的自然演进。提示词工程是“写好指令”，上下文工程是“管理好模型看到的全部信息”。具体来说，它涉及对上下文中 token 的选择、排序、注入和维护的策略，以确保模型能够在有限的上下文窗口中高效、准确地利用信息。

## 为什么要管理上下文

随着我们转向构建能够在多次推理轮次和更长时间跨度上运行的智能体，我们需要管理整个上下文状态，包括系统指令、工具、MCP、外部数据和消息历史等。

可以把循环中的智能体看作一个信息生成器：每轮推理都会产生新的数据、工具调用结果、状态变化、观察记录等。这些内容可能对下一轮决策有用，但上下文窗口有限，不能全部保留。

根本原因是“上下文腐化”现象：随着上下文窗口中 token 数量增加，模型从该上下文中准确回忆信息的能力会下降。与工作记忆容量有限的人类类似，LLM 在解析大量上下文信息时，会动用其注意力预算。此外，模型从训练数据分布中形成其注意力模式，而训练数据中短序列通常比长序列更常见。

因此，深思熟虑的上下文工程对于构建强大的智能体至关重要。

## 好的上下文

上下文工程不是给模型喂得越多越好，而是如何在有限注意力资源下，为模型构造最小、最干净、最有决策价值的信息环境。

好的上下文可以理解为：

- 任务目标
- 当前状态
- 关键证据
- 约束条件
- 下一步所需信息

上下文工程要解决的是：给模型看什么、不看什么、以什么形式看、什么时候更新、旧信息如何压缩、新信息如何进入。

## 系统提示词

系统提示词的艺术，是不写死具体分支，但给出清晰的行为原则；不只讲抽象目标，而给出可执行的判断标准。

系统提示词应该提供决策框架，而不是提供死流程。可以把系统提示词分成三层：

- 身份和目标：你是什么智能体，要完成什么类型的任务。
- 行为原则：遇到任务时应该如何判断、如何取舍、如何处理不确定性。
- 边界约束：什么不能做，什么时候要确认，什么时候要使用工具，什么时候要停止。

好的系统提示词通常应该重点写第二层和第三层。系统提示词要结构化，但不要过度堆料；要从最小可用版本开始，通过测试失败案例逐步补强。

系统提示词是一套行为约束和决策启发式，应当结构清晰、信息最小充分，并通过真实任务中的失败模式持续迭代。

## 工具与示例

工具是智能体上下文工程的核心组件。智能体通过工具和外部世界交互。每一次工具调用，都会产生新的上下文：搜索结果、文件内容、数据库记录、命令输出、错误日志、API 返回值等。

工具不仅执行动作，还会向上下文窗口注入信息。工具返回结果必须是高信号、低噪声的信息。好的工具输出应该是“足以支持下一步推理”的压缩信息。

工具之间的边界要清晰。一个好的智能体不一定需要很多工具，而是需要一个最小可行的工具集，这和函数设计哲学很像。

少样本示例也很重要，但不能把提示词变成边缘案例大全。更好的做法是：选择少量、多样、规范的示例。示例的价值不是列举规则，而是给模型一个行为模式。

系统提示词提供行为框架；工具定义行动边界；示例塑造行为模式；消息历史维持任务状态。上下文工程就是把这些组件组织成一个最小、清晰、高信号的决策环境。

## 长周期任务

Agent 就是在循环中自主使用工具的 LLM。每次交互都会产生新的上下文，用于下一步决策。智能体只在工作记忆中维护必要信息，此为渐进式披露。

随着 LLM 能力提升，智能体设计将更多依赖模型自主决策、减少人工策划。在 Claude 架构上，最优策略通常是“最简单且有效的方案”：明确上下文边界、合理利用即时检索和工具组合，同时保留可扩展性与渐进式探索能力。

长周期任务要求智能体在一系列行动中保持连贯性、上下文意识和目标导向行为，而这些行动所产生的 token 数量可能超过 LLM 的上下文窗口。因此，长周期智能体的核心工程问题，就是持续压缩、筛选、重组和更新上下文。

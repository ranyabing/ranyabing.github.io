## 智能体和工作流的区分

工作流的流程是预定义好的，决策已经被写好，比如遇到某个步骤执行某个操作，都是事先写好的。

智能体是动态的，一切路径和操作由 LLM 决策，开发者只需要为 LLM 搭建一套 harness，核心是 LLM 的智能。

Anthropic 内部把工作流和智能体统称为智能体系统（Agentic Systems）。

- 工作流是通过预定义的代码路径来协调 LLM 和工具的系统。
- 智能体则是 LLM 动态地指导其自身流程和工具使用的系统，并保持对其如何完成任务的控制。

## 何时选择智能体

当任务非常简单，尤其是提示词工程或者接一个 RAG 就可以解决问题时，不需要搭建智能体系统。

当任务需要更高复杂性时，如果问题定义非常明确，而且对稳定性和一致性要求高，那选择工作流。

当需要灵活性和模型驱动的决策时，选择智能体。

对于绝大部分 LLM 应用，使用 RAG 和上下文示例优化单个 LLM 调用就足够了。

## 何时使用框架

比如 LangGraph，使用框架的好处是简化了标准的底层任务，入门容易。

但是框架会创建额外的抽象层。如果不理解框架底层代码，这会让调试变得困难。

因此，推荐一开始从 LLM API 搭建，许多模式可以用几行代码实现。如果确实要使用框架，需要确保了解框架的底层代码。

## 构建块

智能体系统的基本构建模块，是使用增强功能增强的 LLM，例如检索、工具和记忆。这些增强功能可以使用模型上下文协议 MCP 来统一构建。

## 工作流模式

### 提示链

提示链将任务分解为一系列步骤，其中每个 LLM 调用处理前一个调用的输出。你可以在任何中间步骤添加门控机制，以确保流程仍在正轨上。

这种方式适合任务能够被明确分解为固定子任务的情况，主要目标是通过使每个 LLM 调用成为一个更简单的任务，来权衡延迟以获得更高的准确性。

### 路由

路由对输入进行分类，并将其定向到专门的后续任务。此工作流程允许关注点分离，并构建更专业的提示。

路由适用于存在最好单独处理的不同类别，并且可以通过 LLM 或更传统的分类模型准确处理分类的复杂任务。

### 并行化

LLM 有时可以同时处理一项任务，并以编程方式聚合它们的输出。当可以并行划分子任务以提高速度，或者当需要多个视角或尝试以获得更高置信度结果时，并行化是有效的。

并行化有两种变体：

- 分段：将任务分解为并行运行的独立子任务。
- 投票：多次运行同一任务以获得不同的输出。

### 协调者-工作者

在协调者-工作者工作流程中，中央 LLM 动态地分解任务，将其委派给工作者 LLM，并综合它们的结果。

此工作流程非常适合无法预测所需子任务的复杂任务。虽然在形式上相似，但与并行化的关键区别在于其灵活性：子任务不是预定义的，而是由协调者根据特定输入确定的。

### 评估者-优化者

在评估者-优化者工作流程中，一个 LLM 调用生成响应，而另一个 LLM 调用在循环中提供评估和反馈。

当我们有明确的评估标准，并且迭代改进提供可衡量的价值时，此工作流程尤其有效。文学翻译很适合这个模式，借鉴文学评论的机制。

两个适用迹象是：人类清楚表达反馈时，LLM 响应可以得到明显改进；LLM 也可以提供这类反馈。

## 智能体

智能体出现的原因，是 LLM 在理解复杂输入、进行推理和规划、可靠地使用工具以及从错误中恢复这些方面越来越强。因此，智能体的智能一定是来自模型本身。

智能体以人类用户的命令或交互式讨论开始其工作。任务明确后，智能体会独立计划和操作，可能会返回给人类以获取更多信息或判断。

在执行过程中，智能体必须在每个步骤从环境中获得真实信息以评估其进度，例如工具调用结果或代码执行结果。然后，智能体可以在检查点或遇到障碍时暂停以获取人工反馈。

智能体可用于难以或不可能预测所需步骤数量，并且无法硬编码固定路径的开放性问题。LLM 可能会运行很多轮，你必须对其决策制定有一定的信任度。

智能体的自主性意味着更高的成本，以及潜在的累积错误。因此，应该在沙盒环境中进行广泛测试，并采取适当保护措施。

## 设计哲学

奥卡姆剃刀原理：仅在复杂性能够明显改善结果时，才考虑增加复杂性。
